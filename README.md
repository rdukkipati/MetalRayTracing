# Ray Tracing in One Weekend (Using Metal Graphics API)

An implementation of Ray Tracing in One Weekend by Peter Shirley (https://raytracing.github.io/books/RayTracingInOneWeekend.html). This time, I'm using the Metal framework from apple to accelerate the ray tracing so I can produce larger images. In some ways, it was exactly what I wanted. I was able to easily make images that covered my entire macbook's screen. However, as I upped the sample count and max ray bounces, I noticed the flaws with this code's approach. At the time of writing this code, I was not yet fully aware of the implications of thread groups on the performance of algorithms on GPU architecture. Thread groups are a hardware simplification that allow for a single instruction decoder to be shared by a group of threads instead of having one for each thread, among other things. However, this comes at a cost. Threads achieve greater performance when all the threads in their group follow the same execution path. If threads within a group diverge from other threads in the same group, some of the threads wait while the other threads execute their instruction. This code features a single massive shader that does all the ray bounces for every pixel on the final image. This results in extreme divergence that compounds the more ray bounces and samples I add. As a result, with higher ray bounces and sample counts, any potential performance gains from using a GPU are lost.

My next attempt at ray tracing will use a wavefront approach similar to the one described in Physically Based Rendering. A wavefront approach utilizes multiple smaller compute kernels that operate on queues of rays. Each kernel only deals with one ray bounce at most, and they only deal with certain types of rays. This approach takes care of the severe divergence issue. It doesn't come without a cost. There's a lot more memory traffic because data is transferred to and from the GPU for each shader invocation. However, the gains from reducing divergence are greater.
